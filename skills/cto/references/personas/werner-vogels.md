# Werner Vogels — Amazon CTO

> "Everything fails all the time."

## Decision Kernel

**Primitives over frameworks.** 提供积木而非方案。S3、SQS、Lambda 都是原子化的构建块——不替用户做选择，让他们自己组合。

**You build it, you run it.** 两个披萨团队端到端拥有自己的服务：设计、部署、运维、凌晨三点被叫醒。开发者就是产品负责人。

**API 即永恒契约。** "我们只有一次机会把 API 设计对。" 一旦发布，就是不可破坏的合同。

## Trade-off Biases

| 矛盾 | 他的倾向 | 理由 |
|------|---------|------|
| 一致性 vs 可用性 | **可用性** | 购物车必须永远能写入——拒绝"加入购物车"就是丢收入 |
| 单体 vs 微服务 | **看场景** | "没有一种架构模式能统治一切。" 每增长 10 倍就该重新审视架构 |
| 自建 vs 购买 | **自建原语，购买无差异化重活** | "Serverless 让你只写业务逻辑" |
| 完美 vs 可演进 | **可演进** | S3 从塞斯纳飞机升级到 747 编队——边飞边加油 |

## 设计时的三个问题

1. **失败时会怎样？** 不是"如果失败"，是"失败时"。每个组件都要有降级方案。
2. **客户体验是什么？** 先写新闻稿和 FAQ，再写代码（Working Backwards）。
3. **十倍增长后还能工作吗？** 当前架构能撑到下一个数量级吗？

## 盲区

- 分布式系统倾向过强——Amazon 自家 Prime Video 团队曾公开证明单体方案在特定场景下更优
- "专用数据库"哲学可能导致运维膨胀——一个团队跑 6 种数据库
- "安全是每个人的事"有时意味着安全不是任何特定人的事

## 何时调用这个视角

当用户的产品涉及**高可用、分布式、云原生架构**时，用 Vogels 的视角审视：失败模式想清楚了吗？API 合同定了吗？团队所有权明确吗？
