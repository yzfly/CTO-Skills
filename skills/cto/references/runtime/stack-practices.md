# Practices Reference — Bridge Layer

This file is the **bridge** between the CTO conversation and the bundled coding/engineering practices in `../payload/`. Load it after `arch.md` §3 (tech stack) is drafted.

**Your job as CTO**: read the bridge sections matching the user's stack, let them inform `arch.md` §5/§6/§8 and the feature specs, then bundle the matching `payload/<category>/` directories into the user's project.

**Never load the full payload files into the conversation.** They're for the downstream coding agent.

```
references/
├── runtime/stack-practices.md      ← you are here
└── payload/                        ← bundle, don't read
    ├── standards/                    (6 language style guides)
    ├── frontend/{react-nextjs, vue-nuxt}/
    ├── backend/{golang, python-fastapi}/
    ├── design/{api, database, system}/
    ├── review/                       (Google code review)
    ├── security/                     (13 SAST vulnerability skills)
    └── methods/                      (17 engineering method skills)
```

Each bridge section follows the same skeleton:
- **Architectural patterns → `arch.md` §5** — patterns to bake in
- **Cross-cutting concerns → `arch.md` §6** — caching, errors, security
- **Prohibitions → `arch.md` §8** — what NOT to do
- **Feature spec adjustments** — which spec sections shift

## Part 1 — Frontend stacks

### React + Next.js

Source: `../payload/frontend/react-nextjs/` (Vercel Engineering, 45 rules + 52 sub-rule files).

**Architectural patterns → `arch.md` §5**

- **Server-first rendering.** Default to React Server Components (RSC). Client components (`"use client"`) are exceptions. Server can import client; client cannot import server.
- **Streaming over blocking.** Pages stream via Suspense boundaries. Each page layout identifies independently-renderable sections.
- **Parallel data fetching by default.** Multiple data sources fetched in parallel (`Promise.all`, co-located server components). Sequential fetches must be deliberate and documented.
- **Client-server boundary as a design surface.** The RSC boundary is the most important architectural seam. Pass IDs and primitives, not full objects.

**Cross-cutting concerns → `arch.md` §6**

- **Caching, three layers.** Per-request: `React.cache()`. Cross-request: LRU / `unstable_cache`. Client: SWR or React Query.
- **Bundle budget.** Analytics/logging load after hydration. Heavy UI via `next/dynamic`. No barrel-file re-exports.
- **Non-blocking side effects.** Emails, analytics, indexing don't block the response. Use `after()` or a queue.

**Prohibitions → `arch.md` §8**

- No sequential awaits for independent data
- No barrel-file imports in client bundles
- No synchronous third-party scripts in critical path
- No passing full DB objects across server-client boundary
- No uncontrolled re-renders from whole-store subscriptions

**Feature spec adjustments**

- §3 Domain Model: mark entities server-only vs. cross-to-client
- §5 Behavior: streaming vs. blocking; explicit parallel fetches
- §7 Edge Cases: loading states (skeleton, spinner, streamed partial)

### Vue + Nuxt

Source: `../payload/frontend/vue-nuxt/` (16 rule files covering Nuxt 3 / Vue 3 SSR patterns).

**Architectural patterns → `arch.md` §5**

- **SSR-first with explicit hydration boundaries.** Initial render data flows through `useFetch` / `useAsyncData` (payload-backed). `$fetch` is for event-driven interactions, not initial render.
- **Server vs. shared code separation.** Anything secret-bearing or per-request lives in `server/api/*` or server plugins. Composables that touch secrets do not exist.
- **Module-scoped state is request-unsafe.** Use `useState` and composable factories — never module-level singletons (SSR runs per request, globals leak across requests).
- **Nitro routes as a first-class backend surface.** `server/api/` is real backend code. Treat it like any backend tier — not "frontend glue."

**Cross-cutting concerns → `arch.md` §6**

- **Hydration consistency.** Server-rendered HTML must match client hydration output. Time-dependent values, locale, user-agent branches must be deterministic or wrapped in `<ClientOnly>`.
- **Runtime config split.** `runtimeConfig` = server-only secrets. `runtimeConfig.public` = client-visible (treat as leaked to attackers).
- **Stable async data keys.** Explicit keys for `useFetch` / `useAsyncData`. Same key across components must use consistent `transform` / `pick` / `deep`.
- **Page meta and head.** Use `useSeoMeta` / `useHead` per route. SEO metadata belongs in the route, not in a layout wrapper.

**Prohibitions → `arch.md` §8**

- No secrets in `runtimeConfig.public` or in client-imported composables
- No mutable module-level globals for request-scoped state
- No `$fetch` in setup for initial render data (use `useFetch` / `useAsyncData`)
- No autogenerated keys for shared async-data state
- No locale/time/user-agent branching that breaks hydration

**Feature spec adjustments**

- §3 Domain Model: which fields are SSR-rendered vs. client-only
- §5 Behavior: payload-backed fetch vs. on-interaction fetch
- §6 Interface: which routes are SSR, SPA, or `<ClientOnly>`

## Part 2 — Backend stacks

### Go (Golang)

Source: `../payload/backend/golang/` (samber's 42 sub-skills, MIT). The collection is dense; below is the design-time digest.

**Architectural patterns → `arch.md` §5**

- **Project layout as contract.** Standard project layout (`cmd/`, `internal/`, `pkg/`, `api/`). `internal/` enforces visibility — use it to harden module boundaries.
- **Context-first.** Every request-scoped function takes `ctx context.Context` as first arg. Cancellation, deadlines, and trace propagation flow through it.
- **Concurrency by default, but bounded.** Use `errgroup` / `singleflight` / worker pools — never naked `go func()`. Every goroutine has a known lifecycle and a leak-free shutdown path.
- **Interfaces at consumer side.** Define interfaces where they're used, not where types are defined. Keeps packages decoupled.
- **Dependency injection without magic.** Prefer constructor injection. If a DI framework is needed, use `google/wire` (compile-time) over `uber/fx` (runtime) — fewer surprises.

**Cross-cutting concerns → `arch.md` §6**

- **Error handling as data.** Errors are values, not exceptions. Use `errors.Is` / `errors.As` / `samber/oops` for structured error wrapping. Define sentinel errors per package.
- **Observability via `slog` + OTel.** Structured logging through `log/slog`. Traces and metrics via OpenTelemetry. No `fmt.Println` in production paths.
- **Database with `database/sql` or `sqlc`.** ORMs are discouraged for non-trivial schemas. `sqlc` generates type-safe queries from SQL.
- **Configuration via `viper` + env.** No hardcoded values; secrets injected via env vars or sealed-secret managers.

**Prohibitions → `arch.md` §8**

- No naked `go func()` without lifecycle ownership
- No shared mutable state without sync primitives or channels
- No `panic` for normal control flow
- No `interface{}` / `any` without justification (Go 1.18+ generics first)
- No global state or singletons (constructor injection only)
- No production logs via `fmt.Println` or unstructured `log.Print`

**Feature spec adjustments**

- §3 Domain Model: define struct fields with tags upfront (json, db, validate)
- §5 Behavior: explicit goroutine ownership and shutdown paths
- §6 Interface: gRPC vs. REST decision per service boundary
- §7 Edge Cases: context cancellation, partial failures, retry semantics

### Python + FastAPI

Source: `../payload/backend/python-fastapi/` (zhanymkanov's curated best practices, ~17K stars).

**Architectural patterns → `arch.md` §5**

- **Domain-organized layout, not file-type layout.** `src/{domain}/{router,schemas,models,service,dependencies}.py` per bounded context — not `src/{routers,schemas,models}/`.
- **Async-first throughout.** `async def` endpoints, `AsyncSession` for DB, `httpx.AsyncClient` for outbound. Sync endpoints are exceptions.
- **Pydantic v2 as the contract.** Request/response schemas, settings, validation — all through Pydantic. `.dict()` and `json_encoders` are removed; use `.model_dump()`.
- **Dependencies as composition.** Auth, DB session, current-user, rate-limit — all via `Annotated[T, Depends(...)]`. Routes stay thin.
- **SQLAlchemy 2.0 async.** `async_sessionmaker`, `AsyncSession`. Alembic for migrations.

**Cross-cutting concerns → `arch.md` §6**

- **Settings via `pydantic-settings`.** Domain-scoped `BaseSettings` per module + a global one. Environment-driven, no hardcoded config.
- **Structured exceptions.** Domain-specific exception classes (in `{domain}/exceptions.py`), translated to HTTP responses by exception handlers.
- **Background tasks for non-blocking work.** FastAPI `BackgroundTasks` for in-process; Celery / Arq for cross-process. Decision goes in `arch.md`.
- **Auth via PyJWT (not python-jose).** PyJWT 2.9+ is maintained; python-jose is not.

**Prohibitions → `arch.md` §8**

- No sync endpoints for I/O-bound work
- No file-type layout (`routers/`, `models/`) — use domain layout
- No sync DB drivers; `AsyncSession` only
- No `python-jose` (unmaintained); use `PyJWT`
- No raw `Exception` raises in routes — use domain exceptions + handlers
- No hardcoded settings; everything through `BaseSettings`

**Feature spec adjustments**

- §3 Domain Model: bounded context = directory; Pydantic schema = contract
- §5 Behavior: sync vs. background task per side effect
- §6 Interface: dependency chain (auth → session → service)
- §7 Edge Cases: domain exception → HTTP status mapping

## Part 3 — Cross-cutting design domains

These bridge categories from `../payload/design/` apply regardless of language. Bundle when the user's product touches the domain.

### API Design

Source: `../payload/design/api/` (Microsoft REST API Guidelines, ~25 pattern files).

**When to surface in `arch.md`**: any product with a public, partner, or stable internal API.

**What it changes**:
- §4 Data Flow / §6 Interface: error envelope shape, pagination strategy (cursor vs. offset), filtering syntax (OData-like vs. GraphQL vs. flat), naming conventions (camelCase / snake_case)
- §6 Cross-Cutting: versioning policy (URL vs. header vs. content-type), deprecation timeline, idempotency key contract, evolvable enums
- §8 Prohibitions: no breaking change without version bump; no removing fields without deprecation period; no `null` for missing optional fields (use absent key)

### Database / Schema Design

Source: `../payload/design/database/` (PostgreSQL guides + schema designer + migration architect).

**When to surface in `arch.md`**: every product with persistent state. Mandatory unless the product is genuinely stateless.

**What it changes**:
- §5 Patterns: normalization level (3NF default; denormalize with explicit reason), choice of `text` over `varchar(n)`, `timestamptz` over `timestamp`, `numeric` over `money`
- §6 Cross-Cutting: migration strategy (Alembic / golang-migrate / sqlx-migrate), online migration discipline (no long locks, additive-then-backfill-then-cutover)
- §8 Prohibitions (PostgreSQL specifics): no `SQL_ASCII`, no `NOT IN` (NULL trap), no `BETWEEN` on timestamps, no `char(n)`, no uppercase identifiers, no `money` type, no `serial` (use `identity`)

### System Design Documentation

Source: `../payload/design/system/design-md-spec.md` (Google Labs DESIGN.md spec).

**When to surface**: always, as the format guide for `arch.md` itself. The CTO's `arch.md` template (in `templates.md`) already aligns with this spec — bundle the file as a sanity reference for the coding agent.

## Part 4 — Engineering Methods

These are cross-cutting engineering disciplines from `../payload/methods/` (Addy Osmani's agent-skills). They apply regardless of stack.

### Which methods matter at CTO design time

| Method | When to reference | How it affects CTO output |
|--------|-------------------|--------------------------|
| `spec-driven-development` | Always — this IS the CTO workflow | Validates that your specs match its quality bar |
| `architecture-governance` | When the design crosses 2+ services | Adds ADR cadence, fitness functions to `arch.md` |
| `idea-refine` | During the opening turns when the vision is fuzzy | Sharpens `brief.md` §1 (vision) and §3 (problem) |
| `planning-and-task-breakdown` | At delivery time | Structures the spec → task handoff |
| `documentation-and-adrs` | When recording decisions in `arch.md` §2 | Ensures ADRs have context, alternatives, consequences |
| `performance-optimization` | When user mentions scale, speed, or cost | Informs drivers (§1) and prohibitions (§8) |
| `shipping-and-launch` | At delivery time | Adds pre-launch checklist, monitoring, rollback to `arch.md` §6 |

### Which methods are coding-agent-only

Don't load these in the CTO conversation — list them in the handoff:

| Method | What it does |
|--------|-------------|
| `test-driven-development` | Write tests before code |
| `incremental-implementation` | Land changes in small, working steps |
| `debugging-and-error-recovery` | Systematic root-cause debugging |
| `code-simplification` | Refactor for clarity without behavior change |
| `git-workflow-and-versioning` | Commit, branch, conflict resolution |
| `ci-cd-and-automation` | Pipeline setup and quality gates |
| `browser-testing-with-devtools` | Real browser testing via DevTools |
| `source-driven-development` | Ground implementation in official docs |
| `deprecation-and-migration` | Manage removals and migrations |
| `context-engineering` | Optimize agent context and rules files |

### What to add to `arch.md` from these practices

**§6 Cross-Cutting Concerns — add when relevant:**
- **Deployment strategy** (from `shipping-and-launch`): staged rollout, feature flags, monitoring/alerting, rollback trigger
- **Architecture governance** (from `architecture-governance`): ADR cadence, fitness functions, exception process

**§8 Boundaries — add when relevant:**
- No feature without acceptance criteria (from `spec-driven-development`)
- No untested critical path (from `test-driven-development`)
- No big-bang merges (from `incremental-implementation`)

## Part 7 — Security & SAST

Source: `../payload/security/` (utkusen's 13 SAST vulnerability detection skills + orchestrator).

**When to surface in `arch.md`**: any product with user input, auth, file handling, or third-party integration. (i.e., almost always.)

**What it changes**:
- §6 Cross-Cutting: input validation boundary, auth token transport, secrets management, file upload constraints, SSRF/IDOR/SQLi mitigations applicable to the chosen stack
- §8 Prohibitions: no hardcoded secrets, no unvalidated input crossing trust boundaries, no string-concatenated SQL, no unrestricted file uploads, no open redirects
- Handoff: include `security/sast-claude.md` as the entry point — it orchestrates the 13 vulnerability checks
